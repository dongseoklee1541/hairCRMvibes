# 사용성 후보: 오늘 예약 중심 홈

## 상태
- Candidate (정식 R ID 미배정)
- 평가 시점: R-14 대표 사용자 검증 이후
- 구현 승인: 없음
- 최종 업데이트: 2026-07-13

## 가설
살롱 업무를 시작할 때 가장 먼저 확인하는 정보가 고객 목록보다 오늘 예약이라면, 홈 상단에 오늘 일정을 배치했을 때 탐색 시간과 스크롤이 줄어듭니다.

## 현재 근거
- 현재 `app/page.js`는 `내 고객`을 화면 제목으로 사용하고 고객 검색·고객 목록 다음에 오늘 예약을 표시합니다.
- 오늘 예약 데이터는 이미 KST 오늘 날짜 기준으로 조회되므로, 후보 검토에 필요한 기본 데이터는 존재합니다.
- 다만 실제 사용자가 앱을 열 때 고객 검색과 일정 확인 중 무엇을 더 자주 하는지는 아직 관찰하지 않았습니다.

## 후보 범위
- 홈의 첫 섹션을 오늘 예약 요약으로 구성합니다.
- 현재 시각과 가까운 다음 예약을 눈에 띄게 표시하는 방안을 검토합니다.
- `새 예약`과 `전체 예약 보기`를 글자 라벨이 있는 큰 행동으로 제공합니다.
- 고객 검색과 고객 목록은 같은 홈에서 아래로 이동하거나 명확한 다음 섹션으로 유지합니다.
- 오늘 예약의 loading, error, empty 상태에도 다음 행동을 제공합니다.

## 범위 밖
- 예약 알림·리마인드 발송
- 매출 또는 통계 카드 추가
- 직원별 일정, 다중 지점, drag-and-drop 일정판
- 데이터 모델·RLS·PWA 캐시 변경
- 예약 상세 페이지 신설

## 채택 판단

### 채택 신호
- 대표 사용자가 앱을 연 뒤 오늘 예약을 찾기 위해 반복해서 스크롤합니다.
- 하루 첫 업무와 고객 응대 전 확인 과제에서 일정 확인이 고객 검색보다 자주 수행됩니다.
- 현재 홈에서 다음 예약 시간이나 빈 시간을 놓치는 사례가 관찰됩니다.

### 보류 신호
- 대표 사용자의 첫 행동이 대부분 이름·전화번호 고객 검색입니다.
- 오늘 예약은 하단 탭의 예약 화면만으로 빠르고 확실하게 확인됩니다.
- 홈 순서를 바꾸면 고객 찾기 시간이 오히려 늘어납니다.

R-14 검증에서 채택 신호가 반복될 때만 별도 Implementation Plan과 정식 ID 승인을 요청합니다.

## 승격 시 검토할 UX
- 오늘 날짜는 `lib/dateTime.js`의 KST helper를 그대로 사용합니다.
- 완료·취소 예약을 오늘 요약에 포함할지, 예정 예약과 어떻게 구분할지 설계에서 확정합니다.
- 다음 예약 표시는 현재 시각 이후 confirmed 예약 중 가장 가까운 항목처럼 명시적인 규칙을 사용합니다.
- 예약 카드에는 최소한 시간, 고객명, 시술명, 상태를 읽기 쉬운 순서로 표시합니다.
- 예약이 없을 때 `오늘 예약 추가` 행동을 제공합니다.
- 고객·예약 정보를 브라우저 저장소나 서비스워커 캐시에 새로 저장하지 않습니다.

## 승격 시 완료 기준
- Pencil에서 홈 기본/loading/error/empty/예약 있음 상태를 먼저 설계합니다.
- 390×844와 360×800에서 오늘 예약과 주요 행동이 첫 화면에서 이해 가능하고 가로 overflow가 없습니다.
- 예약 건수가 많아도 스크롤과 하단 탭이 충돌하지 않습니다.
- KST 자정·날짜 변경 후 오늘 데이터가 새로 조회됩니다.
- 대표 사용자 과제에서 오늘 예약 찾기 시간과 불필요한 스크롤이 기존 화면보다 감소합니다.

## 위험
- 고객 검색이 실제 최우선 업무라면 순서 변경이 오히려 불편할 수 있습니다.
- 다음 예약 계산이 현재 시각·상태 규칙과 어긋나면 잘못된 정보를 강조할 수 있습니다.
- 첫 화면에 정보를 많이 넣으면 R-14의 단순화 목표와 충돌할 수 있습니다.

## 승격 규칙
이 문서는 번호를 예약하지 않습니다. 채택이 승인된 시점에 당시의 다음 사용 가능 R 번호와 새 R 문서를 만들고, 이 문서는 결정 근거를 남기는 후보 기록으로 유지합니다.
