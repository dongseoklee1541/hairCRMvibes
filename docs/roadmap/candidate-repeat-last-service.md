# 사용성 후보: 지난 시술 그대로 재예약

## 상태
- Candidate (정식 R ID 미배정)
- 평가 시점: R-14 대표 사용자 검증 이후
- 구현 승인: 없음
- 최종 업데이트: 2026-07-13

## 가설
단골 고객은 같은 시술을 반복하는 경우가 많으므로 고객 이력에서 `같은 시술로 예약`을 시작하면 고객·시술·소요시간 재입력과 선택 실수가 줄어듭니다.

## 현재 근거
- `app/customers/[id]/page.js`는 고객의 예약·시술 이력을 이미 표시합니다.
- `app/appointments/new/page.js`는 고객, 날짜, 시간, 활성 서비스, 소요시간을 새로 선택합니다.
- R-08은 예약 당시 서비스명·가격·소요시간 snapshot과 현재 활성 서비스 마스터를 구분합니다.
- 실제로 같은 시술을 반복하는 비율과 재입력 불편은 아직 측정하지 않았습니다.

## 후보 범위
- 완료된 시술 이력에 `같은 시술로 예약` 행동을 제공합니다.
- 새 예약 화면을 열 때 고객을 미리 선택합니다.
- 과거 서비스가 현재도 활성 상태라면 현재 서비스와 기본 소요시간을 제안합니다.
- 날짜와 시간은 과거 값을 복사하지 않고 반드시 새로 선택하게 합니다.
- 자동 저장하지 않고 사용자가 모든 값을 확인한 뒤 기존 예약 등록 버튼으로 저장합니다.

## 데이터·가격 원칙
- 과거 가격 snapshot을 새 예약 가격으로 조용히 복사하지 않습니다.
- 현재 활성 서비스가 있으면 현재 가격을 명확하게 보여주고 새 예약 snapshot은 저장 시점 값으로 생성합니다.
- 과거 서비스가 비활성화됐거나 연결되지 않으면 이유를 설명하고 다른 활성 시술을 선택하게 합니다.
- 고객 메모와 과거 예약 메모는 자동 복사하지 않습니다. 오래되거나 민감한 내용을 새 예약에 잘못 남기는 것을 방지합니다.
- 고객 ID나 서비스 ID 전달은 최소 범위로 하고 localStorage, IndexedDB, Cache Storage에 저장하지 않습니다.

## 범위 밖
- 반복 예약 자동 생성
- 여러 날짜 일괄 예약
- 과거 가격 보장, 할인, 쿠폰
- SMS·Push 확인 발송
- 서비스 마스터나 예약 schema 변경

## 채택 판단

### 채택 신호
- 같은 고객의 최근 완료 시술과 새 예약 시술이 반복되는 비율이 높습니다.
- 대표 사용자가 고객 상세와 새 예약 화면을 오가며 같은 정보를 다시 찾습니다.
- 시술 또는 소요시간을 잘못 선택해 수정하는 사례가 관찰됩니다.

### 보류 신호
- 고객마다 시술이 자주 달라 과거 값 제안이 혼란을 만듭니다.
- 현재 빠른 고객 검색과 기본 서비스만으로 등록 시간이 충분히 짧습니다.
- 비활성 서비스와 현재 가격 확인이 추가 선택보다 더 복잡해집니다.

R-14 검증에서 반복 입력 문제가 확인되고 운영 데이터의 비식별 집계 방법이 승인될 때만 정식 승격을 검토합니다.

## 승격 시 UX 대안

### A안 - 최근 완료 이력에서 시작
- 사용자가 복사할 이력을 직접 선택하므로 의도가 명확합니다.
- 서비스가 비활성 상태일 때 해당 카드에서 즉시 안내할 수 있습니다.
- 권장안입니다.

### B안 - 고객 선택 뒤 최근 시술 자동 제안
- 동작은 빠르지만 자동 선택을 사용자가 놓칠 수 있고 잘못된 과거 값을 그대로 저장할 위험이 큽니다.
- 기본값으로 자동 확정하지 않고 선택형 제안으로 제한해야 합니다.

## 승격 시 완료 기준
- Pencil에서 활성 서비스 연결, 비활성 서비스, legacy 자유입력 이력, 취소, 저장 오류 상태를 설계합니다.
- 고객은 정확히 유지되고 날짜·시간은 비어 있거나 명시적 새 기본값으로 시작합니다.
- 현재 서비스명·현재 가격·소요시간이 저장 전에 보이며 과거 가격과 혼동되지 않습니다.
- 뒤로 가기와 취소 시 기존 고객 상세 위치로 안전하게 돌아갑니다.
- 390×844와 360×800에서 재예약 시작부터 저장까지 키보드·CTA 가림이 없습니다.
- 대표 사용자 과제에서 동일 시술 예약의 입력 단계 또는 완료 시간이 기존 흐름보다 감소합니다.

## 위험
- 과거 snapshot과 현재 서비스 가격을 혼동하면 가격 분쟁이 생길 수 있습니다.
- 비활성·legacy 서비스 처리까지 고려하지 않으면 일부 고객만 동작합니다.
- 편의를 위해 메모를 복사하면 오래된 개인정보나 특이사항이 잘못 재사용될 수 있습니다.

## 승격 규칙
이 문서는 번호를 예약하지 않습니다. 채택이 승인된 시점에 당시의 다음 사용 가능 R 번호와 새 R 문서를 만들고, 이 문서는 결정 근거를 남기는 후보 기록으로 유지합니다.
