## 1. 검증 목적과 기존 근거

R-14의 대표 사용자 검증 기준은 다음 세 가지 핵심 과제를 최소 2명의 50~60대 여성 대표 사용자가 도움 없이 수행하는지 관찰하고, 각 과제의 막힌 위치·오조작·이해하지 못한 용어·완료 확신을 기록하는 것이다.

1. 고객 찾기
2. 새 예약 등록
3. 예약 확인 또는 상태 변경

현재 R-14 문서의 상태는 `구현 완료 · 대표 사용자 검증 대기`다. 기존 합성 증거는 다음 범위를 참고하되 실제 사용자 세션의 대체 증거로 사용하지 않는다.

- `output/playwright/r14-easy-usability/`: 390×844·360×800 정상 화면, 상태 화면, 키보드 축소, 오프라인 fallback, Pencil 검증 PNG
- 기존 합성 브라우저 검증: 가로 overflow 0, 44×44px 미만 상호작용 요소 0, 의도적으로 주입한 오류 외 console warning/error 0
- 기존 PWA 검증: 서비스워커 activated/controlled, precache 내 Auth·고객·예약 민감 cache entry 0개, 오프라인 fallback 확인
- 기존 검증의 미완료 항목: 실제 50~60대 여성 관찰, 모바일 실기기 IME, standalone, 서비스워커 update

기존 캡처와 합성 결과는 참가자에게 보여 주어 행동을 유도하거나 정답을 설명하는 자료로 사용하지 않는다.

## 2. 운영 원칙

- 나이와 성별을 능력의 대리값으로 해석하지 않는다. 관찰 대상은 연령대가 아니라 실제 업무 맥락과 과제 수행 행동이다.
- 진행자는 성공 방법을 알려 주지 않는다. 참가자가 무엇을 하려는지, 어디에서 멈췄는지, 어떤 말을 사용했는지를 기록한다.
- 빠른 완료보다 참가자의 자연스러운 탐색과 판단을 우선한다.
- 참가자가 말한 내용과 진행자가 해석한 내용을 분리한다. 원문 발언은 가능한 한 그대로 기록한다.
- 시스템상 완료와 참가자가 느끼는 완료 확신을 별도 값으로 기록한다.
- 참가자 2명의 결과만으로 일반 사용자 전체를 대표한다고 표현하지 않는다.
- 이 세션에서는 후보 3개를 채택·구현하지 않는다. 결과는 후보 검토를 위한 관찰 근거로만 사용한다.
- 운영 고객·운영 예약·실제 전화번호·실제 메모·실제 계정은 어떤 단계에서도 사용하지 않는다.

## 3. 참가자 조건과 동의

### 3.1 참가자 조건

두 참가자 모두 다음 기준을 세션 시작 전에 확인한다.

- 세션 시점 만 50~60대 여성
- 살롱 운영, 고객 응대, 예약 관리 또는 이에 가까운 업무 맥락을 알고 있거나 대표 사용자로 검토할 합리적인 이유가 있음
- 스마트폰을 본인이 평소 사용하는 방식으로 조작할 수 있음
- 안경, 확대, 보청기 등 평소 사용하는 보조 수단은 그대로 사용 가능
- 디지털 숙련도는 선별 기준이나 평가 대상이 아님
- 구현자, 진행 중인 R-10 담당자, 직접적인 평가 이해관계자는 참가자로 섭외하지 않음

참가자 식별자는 `P01`, `P02`만 사용한다. 이름, 전화번호, 주소, 살롱명, 고객명, 계정 이메일은 관찰 기록에 적지 않는다.

### 3.2 시작 전 설명문

진행자는 다음 취지로 설명한다. 필요한 경우 문장을 그대로 읽는다.

> 오늘은 서비스를 평가하는 것이 아니라 화면이 얼마나 이해하기 쉬운지 알아보기 위한 시간입니다. 잘못 누르셔도 괜찮고, 중간에 그만두셔도 됩니다. 정답을 맞히는 시험이 아니며, 진행자가 화면 조작 방법을 알려 드리지는 않습니다. 실제 고객 정보는 사용하지 않고 모두 연습용 정보만 사용합니다.

### 3.3 동의 확인

세션 시작 전 아래 항목을 말로 확인하고 체크한다.

| 항목 | 확인 |
| --- | --- |
| 관찰 목적과 진행 방식을 설명함 | [ ] |
| 도움을 요청하거나 언제든 중단할 수 있음을 설명함 | [ ] |
| 실제 고객 정보가 아닌 합성 데이터만 사용함을 설명함 | [ ] |
| 화면·음성·영상 기록 여부와 보관 기간을 설명함 | [ ] |
| 기록에 참가자 이름·연락처를 남기지 않음을 설명함 | [ ] |
| 질문할 기회와 동의 철회 방법을 안내함 | [ ] |

화면 녹화나 음성 녹음은 기본값을 `하지 않음`으로 한다. 꼭 필요한 경우에도 별도 동의를 받고, 합성 데이터만 보이는지 확인한 뒤 최소 범위로 기록한다. 동의를 거부해도 세션 참여를 불이익 없이 중단할 수 있어야 한다.

### 3.4 중단·철회 절차

다음 중 하나라도 발생하면 진행자는 즉시 해당 과제를 멈추고 참가자에게 선택권을 준다.

- 참가자가 피로, 불편, 불안, 개인정보 노출 우려를 말함
- 실제 고객 정보나 실제 계정이 화면·자동완성·알림·클립보드에서 나타남
- synthetic 환경인지 확인할 수 없거나 운영 Supabase/Production URL로 연결됨
- 화면 녹화·브라우저 로그·다운로드에 개인정보가 포함될 가능성이 있음
- 반복 오류, 네트워크 장애, 기기 과열 등으로 자연스러운 수행이 불가능함
- 참가자가 명시적으로 중단·철회를 요청함

중단 시 진행자는 다음 문구만 사용한다.

> 여기서 멈추겠습니다. 잘못하신 것이 아니고, 환경을 확인한 뒤 다시 진행할지 결정하겠습니다.

중단 후에는 스크린샷·녹화·로그를 추가로 만들지 않는다. 이미 실제 정보가 노출되었다면 해당 세션을 폐기하고, 아래 개인정보 사고 절차를 실행한다.

## 4. Synthetic 환경 안전 게이트

모든 항목을 통과하기 전에는 참가자에게 기기를 건네지 않는다.

### 4.1 환경·계정 게이트

| 확인 항목 | 통과 기준 | 확인 |
| --- | --- | --- |
| 앱 origin | 로컬 또는 별도 Preview synthetic origin이며 Production origin이 아님 | [ ] |
| Supabase 연결 | synthetic 전용 프로젝트/로컬 인스턴스이며 운영 프로젝트가 아님 | [ ] |
| 환경 변수 | URL·키의 실제 값은 기록하지 않고, 대상 프로젝트가 synthetic인지 화면으로만 확인 | [ ] |
| 인증 계정 | 테스트 전용 계정이며 운영 사용자·참가자 개인 계정이 아님 | [ ] |
| 새 브라우저 컨텍스트 | 이전 세션의 쿠키·자동완성·클립보드·다운로드가 없음 | [ ] |
| 권한 | synthetic owner/staff 중 사전에 정한 최소 권한만 사용 | [ ] |
| 원격 연결 | 운영 API·Production Supabase·실제 고객 데이터 endpoint로의 요청이 없음 | [ ] |
| 초기화 | 참가자마다 fixture를 새로 준비하거나 이전 변경을 완전히 되돌림 | [ ] |

운영 URL과 synthetic URL을 진행자 화면에 동시에 열어 두지 않는다. URL, 프로젝트명, 계정 이메일, 토큰, anon key, 쿠키 값은 관찰 기록과 캡처에 남기지 않는다.

### 4.2 합성 fixture 게이트

세션마다 다음처럼 식별 가능한 합성 fixture를 준비한다. 구체적인 날짜와 시간은 세션 당일 synthetic 환경에서 확인한 유효한 값으로 기록하되, 실제 일정과 일치시키지 않는다.

| 데이터 | 합성 규칙 | 사용 목적 |
| --- | --- | --- |
| 고객 A | `합성 고객 A`처럼 실제 인물과 무관한 이름 | 고객 찾기, 새 예약 등록 |
| 고객 B | `합성 고객 B`처럼 고객 A와 구별되는 이름 | 검색 결과 구별·오조작 관찰 |
| 연락처 | 비워 두거나 서비스가 허용하는 명시적 비실재 test token만 사용. 실제 전화번호 형식은 사용하지 않음 | 연락처 자동완성·실수 입력 방지 |
| 시술 | `합성 시술 A` 등 식별 가능한 synthetic 서비스 | 예약 등록 |
| 날짜·시간 | synthetic 환경의 다음 유효 영업일·충돌 없는 시간 | 예약 등록·예약 확인 |
| 메모 | 실제 생활 정보가 아닌 `합성 메모` | IME 입력·저장 확인이 필요한 경우 |
| 상태 | 기본 `예약`/`confirmed`, 상태 변경 변형은 `완료`/`completed` | 예약 상태 변경 |

fixture에 실제 고객처럼 보이는 이름, 실제 전화번호, 실제 메모, 실제 살롱명, 실제 예약 이력을 넣지 않는다. 참가자가 자신의 고객 정보를 말하더라도 기록하지 않고, 합성 화면으로 돌아온다.

## 5. 진행자 역할과 중립 문구

### 5.1 기본 진행 문구

세션 시작:

> 평소 스마트폰을 사용하시는 방식대로 해 보시면 됩니다. 지금 무엇을 하려고 하는지 말해 주셔도 되고 조용히 진행하셔도 괜찮습니다.

과제 제시:

> 지금부터 이 화면에서 다음 일을 해 주세요. 어디를 눌러야 하는지는 정해 드리지 않겠습니다.

생각을 묻는 말:

> 지금 무엇을 하려고 하셨나요?

> 이 화면에서 다음에 무엇이 될 것 같다고 생각하시나요?

> 방금 보신 문구나 버튼은 어떤 뜻으로 이해하셨나요?

막힘을 확인하는 말:

> 어느 부분에서 멈추셨나요?

> 지금 기대하신 결과와 실제 화면은 어떻게 다른가요?

도움 요청을 받은 뒤:

> 어떤 점이 어려운지 먼저 말씀해 주시겠어요? 원하시면 과제를 중단하거나 준비된 도움을 받을 수 있습니다.

완료 확인:

> 이 과제가 끝났다고 생각하시나요? 그렇게 생각한 근거는 무엇인가요?

종료:

> 오늘 기록은 화면을 더 이해하기 쉽게 만들기 위한 자료로만 사용합니다. 어느 답도 맞거나 틀린 것이 아니며, 더 말씀하고 싶지 않은 내용은 답하지 않으셔도 됩니다.

### 5.2 금지 문구와 대체 문구

| 금지되는 유도 | 대체 문구 |
| --- | --- |
| “여기 버튼을 누르시면 돼요.” | “지금 하려는 일을 계속해 보시겠어요?” |

| “잘하셨어요. 맞아요.” | “현재 화면에서 무엇이 완료되었다고 판단하셨나요?” |
| “조금만 더 눌러 보세요.” | “계속 진행할지, 도움을 받을지 선택하셔도 됩니다.” |

### 5.3 도움 제공 단계

도움은 과제 완료 여부와 별도로 기록한다. 참가자가 명시적으로 도움을 요청하기 전에는 진행자가 화면을 가리키거나 조작하지 않는다.

1. 10초 동안 기다리고 관찰한다.
2. 중립적으로 `지금 무엇을 하려고 하셨나요?`라고 한 번 묻는다.
3. 참가자가 계속 막혔거나 직접 도움을 요청하면 과제 문장을 그대로 다시 읽는다.
4. 그래도 진행할 수 없고 참가자가 원하면 과제를 중단한다.
5. 진행자가 특정 버튼, 메뉴, 입력값, 순서를 알려 주거나 대신 조작했다면 `도움 후 완료`로 기록하고 도움 없이 완료로 세지 않는다.

기술 장애를 해결하기 위한 조작은 사용자 이해도 결과와 분리해 `환경 지원`으로 기록한다. 진행자는 해결 과정에서 제품 사용법을 설명하지 않는다.

## 6. 세션 시간과 공통 시작 조건

- 전체 권장 시간: 30~40분
- 동의·환경 설명: 최대 5분
- 과제 1: 최대 5분
- 과제 2: 최대 8분
- 과제 3: 선택한 변형에 따라 최대 5분
- 과제 사이 회복·fixture reset: 과제당 최대 2분
- 종료 인터뷰: 최대 7분

각 과제의 최대 시간이 지나면 진행자는 다음 문구로 멈춘다.

> 이 과제는 여기서 멈추겠습니다. 지금까지의 과정을 기록하고 다음 과제로 넘어가도 괜찮습니다.

공통 시작 조건:

- 기기는 참가자가 평소 사용하는 방식으로 잡을 수 있는 실제 스마트폰이다.
- 테스트 브라우저 컨텍스트는 새로 시작했고 synthetic 인증 상태만 준비되어 있다.
- 홈 화면 또는 사전에 정한 시작 화면만 열어 둔다.
- 진행자는 과제 시작 전에 메뉴 위치, 용어 뜻, 정답 경로를 설명하지 않는다.
- 각 과제 시작 시 시작 시각을 기록하고, 완료 선언·중단·시간 초과 시각을 기록한다.
- 동일한 과제 변형은 P01과 P02에 동일하게 적용한다.
- 날짜·시간·시술값은 과제 카드에 적어 두되, 참가자에게 앱 안의 어느 위치에 있는지 알려 주지 않는다.

## 7. 과제 카드

과제 카드는 아래 문장만 읽는다. 대괄호 안 값은 세션 전에 synthetic fixture와 대조해 채우며, 참가자에게 경로를 설명하는 메모는 카드에 적지 않는다.

### 과제 1 — 고객 찾기

**시작 조건**

- synthetic 홈 화면 또는 사전에 정한 첫 화면
- `합성 고객 A`와 `합성 고객 B`가 검색 가능한 상태
- 이전 과제의 입력·검색·선택 상태가 초기화됨

**참가자에게 읽을 문장**

> `합성 고객 A` 고객을 찾아서 그 고객의 정보를 화면에 보여 주세요.

**완료 기준**

- 참가자가 `합성 고객 A`의 화면 또는 상세 정보를 스스로 찾아 표시함
- 참가자가 해당 고객을 찾았다고 선언함
- 잘못된 고객 B나 검색 결과를 잠시 열었다가 되돌아온 경우는 오조작으로 기록하되 최종 완료는 인정함
- 진행자가 경로·버튼·용어를 알려 주었다면 도움 후 완료로 기록함

**최대 시간**: 5분

**관찰 포인트**

- 고객 메뉴, 검색 입력, 검색 결과, 고객 상세의 구분
- 검색어를 어떻게 정하는지
- 결과가 없거나 여러 개일 때의 판단
- 이름·아이콘·보조 문구·뒤로 가기의 의미 이해

### 과제 2 — 새 예약 등록

**시작 조건**

- 새 브라우저 컨텍스트 또는 초기화된 synthetic 상태
- `합성 고객 A`가 예약 선택 대상임
- 유효한 synthetic 서비스·날짜·시간이 준비됨
- 준비된 값: 고객 `[합성 고객 A]`, 서비스 `[합성 시술 A]`, 날짜 `[[synthetic 날짜]]`, 시간 `[[synthetic 시간]]`
- 해당 시간에 충돌 예약이 없고 synthetic 환경에서 저장 가능한 상태

**참가자에게 읽을 문장**

> `합성 고객 A`의 다음 방문 예약을 하나 등록해 주세요. 준비된 카드의 서비스, 날짜, 시간으로 진행해 주세요.

필요한 경우에만 값 카드를 다시 보여 준다. 진행자가 입력 순서나 메뉴 위치를 말하지 않는다.

**완료 기준**

- 참가자가 준비된 고객·서비스·날짜·시간으로 저장을 시도함
- 저장 성공 문구 또는 예약 목록·상세에서 새 예약이 확인됨
- 참가자가 예약 등록이 끝났다고 선언함
- 저장 실패가 발생하면 실패 원인과 참가자의 다음 행동을 기록하고, 진행자가 대신 저장하지 않음

**최대 시간**: 8분

**관찰 포인트**

- 고객 선택과 새 고객 등록의 구분
- `서비스 마스터`, `DB`, `snapshot` 등 내부 용어가 노출될 때의 이해
- 필수값·선택값과 저장 버튼의 활성화 의미
- 날짜·시간·시술 선택과 오류 메시지의 연결
- 저장 중·저장 완료·실패 상태의 구분

### 과제 3 — 예약 확인 또는 상태 변경

세션 전에 아래 두 변형 중 하나를 선택하고 P01·P02에게 같은 변형을 적용한다. 기본 변형은 현재 앱에서 지원되는 상태 변경이며, 안전한 synthetic fixture를 준비하기 어려운 경우에만 확인 변형을 사용한다. 선택한 변형은 결과표에 기록한다.

#### 변형 A: 예약 상태 변경(기본)

**시작 조건**

- 과제 2에서 만든 synthetic 예약 또는 동일한 synthetic 예약이 목록에 존재함
- 대상 예약의 초기 상태가 `예약`/`confirmed`임
- 상태 변경 후 되돌릴 수 있는 별도 synthetic fixture 또는 세션 종료 후 reset 절차가 준비됨

**참가자에게 읽을 문장**

> 오늘 일정에서 `합성 고객 A`의 예약을 완료 처리해 주세요.

**완료 기준**

- 대상 예약을 찾아 상태를 `완료`/`completed`로 변경함
- 화면의 상태 배지·메시지·목록 중 하나 이상으로 변경 결과를 확인함
- 참가자가 완료 처리되었다고 판단해 선언함

**최대 시간**: 5분

#### 변형 B: 예약 확인

**시작 조건**

- 대상 synthetic 예약이 예약 목록에 존재함
- 대상 예약 외에 혼동을 관찰할 수 있는 synthetic 예약이 필요하면 최대 1건만 추가함

**참가자에게 읽을 문장**

> `합성 고객 A`의 예약이 제대로 등록되어 있는지 확인해 주세요. 확인했다고 생각하면 어떤 예약인지 말씀해 주세요.

**완료 기준**

- 참가자가 대상 예약의 고객, 날짜, 시간, 서비스 또는 상태를 화면에서 확인함
- 참가자가 확인 결과를 자신의 말로 설명함
- 진행자가 정답을 읽어 주어 확인한 경우는 도움 후 확인으로 기록함

**최대 시간**: 5분

| --- | --- | --- | --- | --- | ---: | --- | --- | --- | --- | --- |
| U-01 |  |  |  | P__ |  |  |  | 없음/부분/실패 |  | 보류 |

| U-03 |  |  |  | P__ |  |  |  | 없음/부분/실패 |  | 보류 |

### 10.1 이슈 분류

- `막힘`: 다음 행동을 선택하지 못했거나 일정 시간 진행을 멈춤
- `오조작`: 의도와 다른 컨트롤·고객·날짜·상태를 선택함
- `용어`: 문구·아이콘·상태를 이해하지 못하거나 다른 뜻으로 설명함
- `확인 부족`: 시스템상 완료됐지만 참가자가 완료 여부를 확신하지 못함
- `환경`: 기기·IME·네트워크·서비스워커 등 제품 사용성 외부 요인
- `안전`: 실제 정보 노출, 운영 데이터 접근, 되돌릴 수 없는 변경 가능성

### 10.2 잠정 판정 규칙

- `검토`: 관찰 근거가 충분하고 후속 제품 논의가 필요한 이슈
- `보류`: 한 번의 선호 표현이거나 과제 완료·이해에 영향이 없어 추가 근거가 필요한 이슈
- `안전 중단`: P0/P1 안전·개인정보·운영 데이터 위험이 있는 이슈

이 표의 `잠정 판정`은 후보 채택이나 구현 승인이 아니다.

## 11. 채택·보류 판단 기준

### 11.1 R-14 대표 사용자 검증 판정

검증 결과를 기록할 때 다음을 모두 충족했는지 확인한다.

- P01과 P02 모두 세 과제의 같은 변형을 수행함
- 각 과제의 시스템상 완료 여부와 참가자 완료 선언을 구분해 기록함
- 핵심 과제마다 도움 없는 완료 여부를 확인함
- 막힌 위치, 오조작, 용어 질문, 완료 확신을 빠짐없이 기록함
- P0/P1 안전 이슈가 없음. 있으면 `통과`가 아니라 `안전 중단/후속 조치 필요`로 기록함
- 실기기 IME, standalone, 서비스워커 update 체크리스트의 수행 여부를 별도로 기록함

한 명이라도 기록이 누락되었거나, synthetic 게이트가 충족되지 않았거나, 도움을 받아야만 완료한 과제가 있으면 `대표 사용자 검증 완료`라고 표현하지 않는다. 그 사실과 남은 작업을 기록한다.

### 11.2 후속 후보 채택·보류 기준

후속 후보 3개는 이 준비 작업에서 채택하지 않는다. 실제 세션 후 다음 기준으로만 `검토` 또는 `보류`를 제안할 수 있다.

`검토` 제안 조건:

- 두 참가자 모두 같은 과제 또는 같은 화면에서 유사한 막힘·오조작·용어 혼동을 보임; 또는
- 한 참가자에게만 발생했더라도 P0/P1 안전·데이터 변경 위험이 확인됨
- 현상이 참가자의 과제 완료, 결과 확인, 잘못된 행동 방지 중 하나에 영향을 줌
- 참가자 원문·화면 위치·시간순 행동이 기록되어 있음

`보류` 조건:

- 한 번의 취향·표현 선호만 있고 과제 결과에 영향이 없음
- 진행자 도움이나 환경 장애 때문에 제품 문제인지 구분되지 않음
- 두 참가자 결과가 서로 다르고 추가 확인 없이는 일반화할 수 없음
- 실제 데이터나 운영 환경이 섞여 근거의 안전성이 깨짐

`검토`는 후보 승격, 코드 수정, 구현을 의미하지 않는다. 후보 3개 중 무엇이든 채택·구현하려면 별도 계획과 승인을 받고, R-14 release record 및 기존 로드맵 문서의 갱신 시점을 다시 확인한다.

## 15. 개인정보 또는 운영 데이터가 나타났을 때의 사고 절차

실제 정보가 한 글자라도 나타나면 다음 순서를 지킨다.

1. 참가자에게 화면을 더 조작하지 않도록 요청하고 세션을 중단한다.
2. 해당 화면의 스크린샷·녹화를 추가로 만들지 않는다.
3. 실제 정보가 포함된 기존 캡처·로그·다운로드를 열람하거나 복사하지 않는다.
4. synthetic 환경과 운영 환경의 origin·계정·네트워크 설정을 확인한다.
5. 가능한 범위에서 세션·브라우저 컨텍스트·테스트 계정을 폐기한다.
6. 기록에는 실제 값을 쓰지 않고 `실제 정보 노출 의심, 세션 중단`만 남긴다.
7. 담당자에게 즉시 보고하고, 원인과 재발 방지 확인 전에는 세션을 재개하지 않는다.

이 사고는 사용자 경험 이슈와 별도의 `P0 안전 이슈`로 집계한다. 데이터 삭제가 확실하지 않으면 대표 사용자 검증 결과를 유효한 근거로 사용하지 않는다.

## 16. 세션 뒤 코드 수정이 필요할 때의 R-10 충돌 감사 Stop Rule

실제 세션 결과를 보고 코드 수정이 필요해 보여도, 아래 감사가 끝나기 전에는 어떤 `app/**`, `components/**`, Supabase, Pencil, R-10 파일도 수정하지 않는다.

### 16.1 즉시 중지

- 관찰 기록과 제품 수정 요청을 분리한다.
- 현재 세션의 증거와 synthetic 상태를 보존하고, 진행 중인 UI를 임의로 고치지 않는다.
- 이 준비 worktree에서는 신규 프로토콜 문서 외 파일을 수정하지 않는다.

### 16.2 먼저 감사할 상태

다음 항목을 읽기 전용으로 확인한다.

- `git worktree list --porcelain`
- 각 관련 worktree의 `git status --short --branch`
- `git branch --all --no-color`
- 열린 PR과 base/head 브랜치
- R-10 작업의 실제 경로와 수정 파일
- 수정하려는 파일과 R-10 수정 파일의 경로·내용·의존성 겹침

현재 준비 작업에서 특히 겹침을 먼저 확인해야 하는 파일은 `components/AppShell.js`, `components/AuthGate.js`, `docs/roadmap/README.md`, `future-todo.md`다. R-10 PR은 `schema.sql`, migration, Pencil, staff API와 서버 권한 코드를 함께 변경하므로, 화면 문제처럼 보여도 데이터·권한·설계 SSOT 변경으로 확대하지 않는다.

### 16.3 중단 조건

다음 중 하나라도 해당하면 사용자에게 감사 결과와 충돌 파일을 보고하고 구현을 중단한다.

- R-10 worktree 또는 브랜치가 같은 `app/**`, `components/**`, `lib/**`, Auth/profile, schema/migration, 설정 파일을 수정 중임
- 동일 파일에 미커밋 변경 또는 열린 PR 변경이 있음
- 현재 checkout과 R-10 worktree가 같은 로컬 파일이나 generated artifact를 공유함
- synthetic 검증 결과를 재현하려고 R-10 worktree를 수정해야 함
- R-10 브랜치를 rebase, stash, reset, checkout, merge, clean 해야 함
- 파일 충돌 여부를 읽기 전용으로 판단할 수 없음

이 경우에는 “수정 가능”으로 추정하지 않는다. R-10은 읽기 전용으로 보존하고, 별도 승인·별도 worktree·충돌 없는 변경 계획이 생길 때까지 멈춘다.

### 16.4 충돌이 없을 때도 필요한 절차

감사상 충돌이 없더라도 실제 코드 수정은 새 Implementation Plan과 승인 후에만 시작한다. `codex/r14-user-validation-prep`에는 신규 프로토콜만 남기고, 코드 수정은 별도 작업 브랜치로 분리한다.

## 17. 결과 보고와 release record 게이트

### 17.1 실제 세션 후 보고 순서

1. 세션 ID, 참가자 코드, synthetic 환경, 기기 정보를 개인정보 없이 기록

### 17.2 문서·PR 순서
